# RAG Codegen AOSP — LLM-based AAOS HAL Code Generation

LLM-based code generation pipeline for Android Automotive OS (AAOS) Vehicle HAL,
using RAG retrieval, DSPy prompt optimisation, and iterative feedback loops.

Generates complete HAL layer code from VSS (Vehicle Signal Specification) signals:
AIDL interfaces, C++ implementations, SELinux policies, Android.bp build files,
design documents, Android app fragments, and backend servers.

**Target AOSP level: Android 14 (API 34)**
All generated code targets the Android 14 Vehicle HAL API surface. Using a different
AOSP version will cause AIDL interface mismatches and build failures.

## Architecture

```
C1-C4: VSS Signals → Labelling → YAML Spec → Module Planner → Code Generation → Validation
                                                                      ↑
                                                  RAG (ChromaDB) + DSPy (MIPROv2)

C5: VTS + HMI Generation (runs on Colab after C4, before Android Build)
    ┌─────────────────────────────────────────────────────────────┐
    │  Input: C4 YAML Spec + MODULE_PLAN.json (Colab outputs)     │
    │         No AOSP build artifacts required                     │
    ├─────────────────────────────────────────────────────────────┤
    │  Agent 1: VTS Generator (RAG+DSPy)                          │
    │    → VtsHalAutomotiveVehicleVss.cpp (tests 500 VSS props)   │
    │  Agent 2: HMI App Generator (C4 DSPy programs reused)       │
    │    → Kotlin fragments + XML layouts with real property IDs   │
    ├─────────────────────────────────────────────────────────────┤
    │  Android Build (GCP VM, after C5):                           │
    │    → VssVehicleHardware service (C3/C4 C++ output)          │
    │    → VTS test binary (C5 output)                             │
    │  Cuttlefish Runtime:                                         │
    │    → VssVehicleHardware serves VSS properties as IVehicle/default │
    │    → VTS tests pass against the running service              │
    └─────────────────────────────────────────────────────────────┘
```

**VHAL Architecture (Android 14 AIDL V3):**
```
VssVehicleHardware : public IVehicleHardware   ← generated by C3/C4 (vendor seam)
DefaultVehicleHal  : public BnVehicle          ← AOSP-owned binder layer
```
C3/C4 generate a from-scratch `IVehicleHardware` implementation that wraps inside
`DefaultVehicleHal`. No `FakeVehicleHardware` patching, no JSON config files.

```
        C1 (baseline LLM)
       /                \
  C2 (+ Thompson)    C3 (+ RAG + DSPy)
                         |
        C4 (C3 + feedback loop)
        Note: Thompson in C4 tracks stats only — not active prompt selection
               ↓
        C5 (VTS + HMI generation on Colab)
          ├── Agent 1: VTS test generation
          └── Agent 2: HMI app (C4 DSPy programs reused)
               ↓
        Android Build (GCP VM)
               ↓
        Cuttlefish VTS (atest VtsHalAutomotiveVehicleVss)
```

**What C5 reuses from C1-C4:**
- All 12 optimised DSPy programs from `dspy_opt/saved/`
- RAG retriever + ChromaDB (same 7 collections)
- `rag_dspy_mixin.py` base class for all agents
- `llm_client.py` Ollama wrapper
- `output_c4_feedback/SPEC_FROM_VSS_*.yaml` and `MODULE_PLAN.json`

**What C5 adds:**
- `VtsGeneratorAgent` — `VtsHalAutomotiveVehicleVss.cpp` testing all 500 VSS property IDs
- `HMI app generator` — Kotlin fragments + XML layouts with real CarPropertyManager IDs
- C5 results reported separately from C1-C4 comparison metrics

### 2. Build RAG Index from AOSP Source

ChromaDB **must** be built before DSPy so the optimizer bootstraps traces with RAG context.
The indexer automatically excludes HIDL files — only AIDL-compatible code is indexed.

> **Do not re-index if ChromaDB already has 7 collections with chunks.**
> Re-indexing without `--force` skips existing collections automatically.
> Use `--force` only if you need to rebuild from scratch.

```bash
# Shallow-clone AOSP repos pinned to android-14.0.0_r75 (~300 MB total)
git clone --depth=1 -b android-14.0.0_r75 \
    https://android.googlesource.com/platform/hardware/interfaces aosp_source/hardware
git clone --depth=1 -b android-14.0.0_r75 \
    https://android.googlesource.com/platform/system/sepolicy     aosp_source/sepolicy
git clone --depth=1 -b android-14.0.0_r75 \
    https://android.googlesource.com/platform/packages/services/Car aosp_source/car

# Index into ChromaDB with AIDL-only filter (~2 min on GPU)
python -m rag.aosp_indexer --source aosp_source --db rag/chroma_db --force
# Expected: 7 collections, AIDL-only (no HIDL/V2_0 content)

# Verify
python -c "
import chromadb
c = chromadb.PersistentClient('rag/chroma_db')
cols = c.list_collections()
print(f'{len(cols)} collections, {sum(col.count() for col in cols)} total chunks')
for col in cols:
    print(f'  {col.name}: {col.count()} chunks')
"
```

# If HIDL contamination or empty output is found, fix the cpp agent
# before spending 4 hours on C4.
python - << 'PYCHECK'
import glob, sys
from pathlib import Path

output_dir = Path("output_rag_dspy")
cpp_files = (list(output_dir.rglob("VssVehicleHardware.cpp")) +
             list(output_dir.rglob("VehicleHalService*.cpp")))

if not cpp_files:
    print("❌ FAIL: No C++ file in output_rag_dspy/ — cpp agent returned empty")
    sys.exit(1)

HIDL_BANNED  = ["HIDL_FETCH_", "hidl/", "Return<", ".valueType", "BnIVehicle",

AIDL_REQUIRED = ["IVehicleHardware", "DefaultVehicleHal", "AServiceManager_addService",
                  "GetValuesCallback", "GetValueRequest", "SetValuesCallback", "SetValueRequest"]
issues = []
for f in cpp_files:
    c = f.read_text(errors="ignore")
    if len(c.strip()) < 200:
        issues.append(f"{f.name}: empty")
    for b in HIDL_BANNED:
        if b in c: issues.append(f"{f.name}: HIDL artifact '{b}'")
    for r in AIDL_REQUIRED:
        if r not in c: issues.append(f"{f.name}: missing '{r}'")

if issues:
    print("❌ C3 cpp AIDL V3 violations:")
    for v in issues: print(f"   • {v}")
    print("Fix before running C4.")
    sys.exit(1)
else:
    print(f"✓ C3 cpp passed AIDL V3 check ({len(cpp_files)} file(s))")
PYCHECK

**Helper functions:**
```bash
# Clean ALL generated files from AOSP tree (AIDL + C++ + SELinux)
# Generated C++ goes into aidl/impl/vss/ — Soong auto-discovers it.
# Use this before re-applying a new C3/C4 run.
restore_aosp() {
    cd ~/aosp-14-auto

    # Remove generated AIDL enum files (keep AOSP originals)
    rm -f hardware/interfaces/automotive/vehicle/aidl/android/hardware/automotive/vehicle/VehicleProperty{Adas,Body,Cabin,Chassis,Hvac,Infotainment,Powertrain}.aidl

    # Remove generated C++ vss/ directory (Soong auto-discovered)
    rm -rf hardware/interfaces/automotive/vehicle/aidl/impl/vss/

    # Remove generated SELinux files
    rm -f system/sepolicy/vendor/vehicle_hal_*.te
    rm -f system/sepolicy/vendor/file_contexts_vss

    # Restore any AOSP files modified during integration
    cd hardware/interfaces && git checkout -- . && cd ~/aosp-14-auto
    cd system/sepolicy    && git checkout -- . && cd ~/aosp-14-auto

    echo "✓ AOSP tree restored"
}
```

### Step 6 — Apply Generated Files and Full Build

`apply_aosp14_fixes.sh` handles all integration — copy-only, no patching.
C3/C4 agents generate correct AIDL output directly; Soong auto-discovers
`aidl/impl/vss/Android.bp` without manual registration.

```bash
cd ~/aosp-14-auto
source build/envsetup.sh
lunch aosp_cf_x86_64_auto-trunk_staging-userdebug

# Full reset (nếu cần, comment nếu không muốn reset mỗi lần)
restore_aosp

# Apply fixes
~/apply_aosp14_fixes.sh ~/output_c4 ~/aosp-14-auto

# Rebuild VSS module
mmm hardware/interfaces/automotive/vehicle/aidl/impl/vss -j$(nproc)

# Update API
m android.hardware.automotive.vehicle-update-api

# Clean VINTF
m clean-vintf
rm -rf out/target/product/vsoc_x86_64_only/vendor/etc/vintf/manifest/android.hardware.automotive.vehicle@V3-vss-service.xml

# Build vendor
m vendorimage -j$(nproc) 2>&1 | tail -30

# Kiểm tra
echo "=== Check VSS VHAL ==="
grep -E "V3-vss-service|vehicle-vss" out/target/product/vsoc_x86_64_only/installed-files-vendor.txt || echo "Not found"

# Push + Mount + Reboot
adb root
adb shell "mount -o remount,rw /vendor"   # ← quan trọng
adb push out/target/product/vsoc_x86_64_only/vendor/bin/hw/android.hardware.automotive.vehicle@V3-vss-service /vendor/bin/hw/
adb reboot
```

`apply_aosp14_fixes.sh` does 4 things (all idempotent):
1. Copy AIDL files → `aidl/android/hardware/automotive/vehicle/`
2. Copy C++ files → `aidl/impl/vss/` + generate `VssVehicleHardware Android.bp`
   (Soong auto-discovers this — no registration in `aidl/Android.bp` needed)
3. Copy SELinux `.te`, VINTF manifest, `init.rc` as-is from C4 output
4. One-time AOSP fixes:
   - Remove conflicting `vintf_fragments` from `vhal/Android.bp` (avoids `IVehicle/default@3` conflict with Cuttlefish emulator service)
   - Remove stale `vhal-default-service.xml` from `out/` if present
   - Unfreeze AIDL interface → `m android.hardware.automotive.vehicle-update-api` (kept unfrozen — sufficient for research use)
   - Add `android.hardware.automotive.vehicle@4` to `fcm_exclude.cpp` (required for types-only AIDL package)

## Known Issues

- **Stale cpp DSPy program:** After the C++ signature change (`ModernCppVehicleHardwareSignature` now uses `domain/properties/aosp_context` inputs and `cpp_impl` output), the saved program at `dspy_opt/saved/cpp_program/` is incompatible. Delete it with `rm -rf dspy_opt/saved/cpp_program/` before running C3/C4. The mixin falls back to unoptimised `ChainOfThought` automatically.
- **Batch labelling mismatch:** LLM returns 1 signal per batch instead of 4; remaining are padded. Prompt/parsing issue in the labelling code.
- **AOSP version:** Generated code targets Android 14 only. Do not use Android 13 or 15 source trees.
- **Do not clone `aosp_source/` inside the AOSP build tree.** Soong scans all directories for `Android.bp` and fails with "module already defined". Fix: `mv ~/aosp-14-auto/aosp_source ~/aosp_source_rag`
- **Use `restore_aosp` for full reset, not `clean_hal`** — `clean_hal` removes only C++ and SELinux generated files. To remove AIDL files too (e.g. before applying a new C3/C4 run), use `restore_aosp` which runs `git checkout .` on `hardware/interfaces` and `system/sepolicy`.
- **Overlapping domain property IDs:** C1-C4 generate per-domain AIDL enums with domain-specific base addresses (ADAS=0x1000, Body=0x2000, etc.) to prevent ID conflicts.
- **SELinux Colab score 0.697–0.747 is expected:** The LLM generates valid AOSP 14 AIDL SELinux using m4 macros (`binder_call`, `hal_server_domain`, `init_daemon_domain`) that standalone `checkpolicy` on Colab cannot expand. The `.te` files compile cleanly on GCP with the full policy tree. No patching is applied — agents generate correct AIDL output directly (`hal_server_domain` instead of HIDL `add_hwservice`).
- **IVehicle/default conflict:** Only one service can register `IVehicle/default`. Stop the stock VHAL (`adb shell stop vendor.vehicle-default`) before starting VssVehicleHardware, or the second `AServiceManager_addService` call fails silently.

## Key Design Decisions

- **IVehicleHardware + DefaultVehicleHal architecture:** All C3/C4 generated C++ implements `IVehicleHardware` (the vendor seam, a plain C++ interface). `DefaultVehicleHal` (AOSP-owned) wraps it as `BnVehicle`. Vendors never subclass `BnVehicle` directly — doing so requires reimplementing async dispatch, subscription management, and shared-memory packaging that DefaultVehicleHal already provides. This is the supported, VTS-passing architecture for Android 14 AIDL V3.
- **AIDL V3 async contract:** `getValues` and `setValues` are asynchronous: they receive a callback (`std::shared_ptr<const GetValuesCallback>`) plus a request vector, invoke the callback with results, then return `StatusCode::OK`. The synchronous `getValues(propIds, areas, out_values*)` form does not exist in V3. All validators and assertions now enforce this contract explicitly.
- **3-layer HIDL defense:** Layer 1 (indexer) — `aosp_indexer.py` excludes HIDL files by path patterns. Layer 2 (mixin) — `_HIDL_CONTAMINATION_PATTERNS` in `rag_dspy_mixin.py` filters HIDL-contaminated RAG chunks at retrieval time (16 patterns: `add_hwservice`, `hwbinder_device`, `fwk_vehicle_hwservice`, etc.). Layer 3 (agent) — each agent signature has explicit FORBIDDEN lists in its docstring; SELinux agent RAG queries target AIDL-only patterns (`hal_server_domain`, `init_daemon_domain`). Stray `{` artifacts from LLM output are stripped in `rag_dspy_mixin._generate()` before `checkpolicy` validation.
- **HIDL exclusion in RAG corpus:** Path-only exclusion (no content filtering) in `aosp_indexer.py`. Content filtering wrongly dropped AIDL reference files (`DefaultVehicleHal.cpp`, `IVehicleHardware.h`) because both HIDL and AIDL contain the same keyword patterns.
- **Hybrid RAG retrieval:** `aosp_retriever.py` uses dense retrieval (all-MiniLM-L6-v2) + BM25 sparse channel + optional cross-encoder reranking. `DEFAULT_TOP_K` raised 3→6 and `max_chars_per_chunk` raised 800→1500 so method signatures are not truncated. The embedding model must stay in sync with the indexer — changing it requires reindexing all 7 collections.
- **Two cpp agent classes:** `rag_dspy_cpp_agent.py` contains `RAGDSPyCppAgent` (thin wrapper, used by the architect and C4 retry engine) and `RagDspyCppAgent` (inner implementation). The architect calls `inner.generate()` directly to get all 4 files (header, impl, main_service, android_bp). C3 no longer has a separate standalone cpp call — the architect handles all 4 files. Both classes share `_QUERIES` and `_CONTRACT` to stay in sync.
- **C4 post-validation architecture:** C4 uses `architect.run()` identically to C3 for initial generation, then post-validates output files and retries only failed agents. Error feedback goes into a separate prompt field to avoid polluting RAG context. C4's floor is always ≥ C3.
- **ChromaDB singleton:** Multiple agents sharing the same ChromaDB path causes "instance already exists" errors. `fix_chroma_singleton.py` monkey-patches `chromadb.PersistentClient` to return a shared singleton.
- **Domain-specific AIDL base addresses:** `DOMAIN_BASE` (ADAS=0x1000, Body=0x2000, Cabin=0x3000, Chassis=0x4000, HVAC=0x5000, Infotainment=0x6000, Powertrain=0x7000) ensures globally unique property IDs across domains. A global running counter in C5 (`global_idx`) prevents collisions when encoding `group|area|type|index`.
- **Version-pinned RAG corpus:** Clone with `android-14.0.0_r75` to match the AOSP build tree. Mismatched versions cause the LLM to generate `Android.bp` patterns and AIDL module structure that don't match the build system.
